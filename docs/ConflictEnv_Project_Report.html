Section 01

Executive Summary

ConflictEnv is an OpenEnv-compatible reinforcement learning environment that simulates the most universally stressful scenario in modern life: managing a chaotic personal calendar under cascading conflicts.


An LLM agent is dropped into the role of a personal executive assistant managing a week-long calendar shared across 7 stakeholders — boss, spouse, clients, doctors, schools, vendors, and airlines. Events overlap, actors push back with counter-proposals, and the underlying APIs (calendar schemas, booking policies, communication formats) drift dynamically between episodes.


What makes ConflictEnv unique is that it naturally unifies all five hackathon themes in a single, coherent environment — without forcing artificial connections. It is simultaneously a multi-agent negotiation arena, a long-horizon planning challenge, a world modeling benchmark, a personal task simulator, and a self-improving curriculum — all through the lens of one relatable human problem.

ConflictEnv is an OpenEnv-compatible RL environment simulating a personal assistant agent managing a realistic weekly calendar with 6–10 actors, where scheduling conflicts cascade and API schemas drift between episodes.

Core Mechanics

🔀 Cascading Conflicts

Moving one event triggers new conflicts. Cancel dentist at 2pm → dentist only has Thursday → Thursday has client call → client needs rescheduling → client's new slot conflicts with kid's recital. 3-5 levels deep.

👤 Actor Negotiation

7 actors with individual preferences, power weights, and tone sensitivity. When the agent reschedules their event, they respond with counter-proposals, acceptance, or rejection — not silent compliance.

🔄 Schema Drift

Every 50 episodes, API schemas mutate: date formats change, field names rename, nested structures appear. The agent must adapt to broken contracts without retraining.

📈 Adaptive Difficulty

When the agent's resolution rate exceeds 70%, the environment auto-injects harder scenarios: more actors, deeper cascades, surprise mid-episode interruptions.

Data Flow

Episode Lifecycle

  1. Reset — Scenario generator creates a calendar with N events, M conflicts, and K actors based on difficulty tier. Schema version is set by episode // 50.
  2. Observe — Agent receives calendar state, conflict list, actor profiles, and policy rules (in the current schema version).
  3. Act — Agent issues one action (reschedule, draft_message, cancel, etc.) with parameters.
  4. Environment Response — Calendar updates, cascading conflicts trigger, affected actors respond with counter-proposals or confirmations.
  5. Repeat — Steps 2-4 for up to max_steps (15 for easy, 25 for medium, 40 for hard).
  6. Terminal — Episode ends when all conflicts resolved, max steps hit, or agent calls resolve. Final reward computed.

API Endpoints (OpenEnv Protocol)

Endpoint Method Input Output

Action Space

Action Parameters Effect Risk
reschedule event_id, new_slot Moves event; may trigger cascade Actor may reject/counter-propose

Section 05

Theme Alignment — All Five, Naturally

ConflictEnv is not five ideas stitched together. It's one coherent scenario where all five themes emerge organically from a single premise.

Wild Card Positioning

"800 teams built environments for one theme. We built one environment that naturally unifies all five — through the most universally relatable problem: managing your own chaotic life."

Theme 1 — Multi-Agent Interactions
7 Actors With Competing Incentives Negotiate Back

ConflictEnv features 7 distinct actors (boss, spouse, client, doctor, school, vendor, airline) each with their own utility function, preferred times, flexibility levels, and tone sensitivity.


When the agent reschedules an event, the affected actor doesn't passively accept. They respond with one of:

  • Accept — "Fine, 10:30 works for me."
  • Counter-propose — "9am doesn't work. I can do 10:30 or push to Tuesday."
  • Reject — "This meeting is locked. Find another way."
  • Escalate — "I need to speak with you directly about this." (penalty trigger)

This creates a genuine cooperative-competitive multi-agent dynamic — actors cooperate on flexible dimensions but compete for priority access to the user's calendar. The agent must model each actor's beliefs, predict their reactions, and negotiate optimal outcomes — exercising theory-of-mind reasoning.


Hackathon fit: Directly satisfies "cooperation, competition, negotiation" requirements. The Halluminate sub-theme ("multi-actor environments where an agent manages multiple actors") is exactly what ConflictEnv does.

Theme 2 — Long-Horizon Planning & Instruction Following
Monday's Fix Breaks Friday — Plan 5 Days Ahead

Conflicts in ConflictEnv cascade across a 5-day horizon. A single reschedule on Monday can trigger a chain reaction that affects Wednesday's client call, Thursday's flight, and Friday's deadline.


The cascade mechanism works in depth:

Cancel dentist at 2pm Monday
  └─→ Dentist only free Thursday 3pm
       └─→ Thursday 3pm = client demo
            └─→ Client pushed to Friday 10am  
                 └─→ Friday 10am = board prep deadline
                      └─→ Board prep needs to happen Thursday night
                           └─→ Thursday night = spouse's birthday dinner

Rewards are sparse — computed only at episode end based on total resolution. There's no intermediate reward signal for each step. The agent must decompose the problem, track state across 25-40 steps, and recover from early mistakes.


Hard scenarios (40 max steps) push beyond shallow next-token reasoning, requiring structured planning and durable internal state tracking.


Hackathon fit: Directly addresses "deep, multi-step reasoning with sparse or delayed rewards" and "decompose goals, track state over extended trajectories."

Theme 3 — World Modeling (Professional + Personal)
Real API Interactions in a Partially Observable World

3.1 Professional Tasks: The agent interacts with simulated but realistic APIs — Calendar API, Travel Booking API, Venue Reservation API. These APIs have proper contract behavior: query → response → side effects. The world is partially observable — the agent doesn't know the boss's current mood, the airline's rebooking fee, or the restaurant's cancellation policy until it queries.


The agent must maintain a consistent internal world model, update beliefs based on API responses, and orchestrate multi-step workflows (e.g., check availability → reschedule → notify actor → confirm).


3.2 Personal Tasks: This IS the core premise. The agent handles exactly the tasks a real personal assistant faces: managing dinner conflicts, replying to tough emails from the boss, handling travel disruptions, negotiating schedule changes with family members.


Schema drift (Patronus AI bonus) makes this world modeling challenge uniquely hard: the API contracts themselves change between episodes. Field names rename, date formats shift, nested structures appear. The agent can't rely on memorized patterns.


Hackathon fit: Covers both 3.1 ("real interaction with tools, APIs, dynamic systems") and 3.2 ("personal task handling, conflicts, delegations"). Schema drift directly targets the Patronus AI sub-theme ("Consumer Workflows with Schema Drift").

Theme 4 — Self-Improvement
The Environment Levels Up When You Do

ConflictEnv implements two self-improvement mechanisms:


1. Schema Drift as Forced Generalization: Every 50 episodes, the environment shifts its schema version (V1 → V2 → V3). An agent that memorized V1's date format fails on V2. Only agents that learned to generalize — to read schema hints, try alternative field paths, gracefully handle parse failures — survive the transitions. This IS self-improvement: the environment forces the agent beyond its current capability ceiling.


2. Adaptive Difficulty Scaling: The environment tracks the agent's rolling Conflict Resolution Rate (CRR). When CRR exceeds 70%, it automatically:

  • Injects more actors into scenarios
  • Deepens cascade chains
  • Adds surprise mid-episode interruptions ("Boss just added an urgent 4pm meeting")
  • Tightens time constraints

The agent effectively faces an adaptive curriculum that escalates as it improves — driving recursive skill amplification rather than plateau on fixed tasks.


Hackathon fit: Satisfies "agents learn to drive their own capability growth" and "adaptive curricula." The Snorkel AI sub-theme ("simulated experts with changing requirements/preferences") is addressed by actors whose preferences shift across episodes.

Theme 5 — Wild Card: Impress Us
First Unified Personal-Agent Benchmark Across All Themes

The wild card play is the meta-narrative itself: no other submission will claim to naturally unify all five themes in a single, coherent environment.


This isn't a Frankenstein of five features bolted together. It's one human problem — managing a chaotic calendar — where multi-agent negotiation, long-horizon planning, world modeling, personal task handling, and self-improvement all emerge organically from the structure of the problem itself.


The creative insight is recognizing that personal scheduling conflicts are the natural intersection point of all five themes. This reframing is the innovation.

Schema Drift Engine — The Differentiator

The single feature that separates ConflictEnv from every other submission. Schema drift is a first-class citizen, not an afterthought.

What Drifts

Dimension V1 (Baseline) V2 (Mild Drift) V3 (Heavy Drift)
Date format "2026-04-25 09:00" "04/25/2026 9:00AM" {"day":25,"month":4,"year":2026,"hour":9}
Calendar key events[].start_time events[].startTime events[].schedule.begin
Cancel policy "free_cancel": true "cancellation_fee": 0 {"policy":{"cancel":{"fee_usd":0,"window_hrs":24}}}
Actor pref actor.preferred_time actor.availability.preferred actor.scheduling_prefs.ideal_slots
Conflict type "type": "overlap" "conflict_kind": "OVERLAP" {"issue":{"category":"TIME_COLLISION"}}
Venue status "available": true "status": "AVAILABLE" {"booking":{"state":"open","slots_remaining":4}}

Drift Schedule

Drift is deterministic: schema_version = episode_number // 50


  • Episodes 0–49 (V1): Agent learns basic conflict resolution with stable schemas
  • Episodes 50–99 (V2): Field renames and format changes — reward drops, then recovers
  • Episodes 100+ (V3): Structural changes — nested objects, new formats — tests true generalization

This creates clean visual signatures in reward curves: steady climb → sharp drop at drift boundary → recovery. Perfect for the demo.

Why Schema Drift Matters

Patronus AI Bonus Prize Alignment

The Patronus AI sub-theme is "Multi-step consumer workflow environments where the underlying data schemas, API contracts, and t&cs/policies/rules change." ConflictEnv implements exactly this — schema drift across calendar APIs, travel booking contracts, and venue policies. This is not a stretch; it's a direct hit.

Section 07

Reward System Design

Multi-Signal Reward Function

reward = (0.40 × conflict_resolution_rate)    # % of conflicts fully resolved
       + (0.30 × stakeholder_satisfaction)     # Mean actor satisfaction (0-1)
       + (0.20 × deadline_adherence)           # Hard deadlines met / total
       + (0.10 × efficiency)                   # 1.0 - (steps / max_steps)

final_reward = clamp(reward, 0.05, 0.95)       # OpenEnv compliant range

Reward Component Breakdown

Component Weight Range What It Measures
Conflict Resolution Rate (CRR) 40% 0.0 – 1.0 How many conflicts were fully resolved vs. total
Stakeholder Satisfaction (SSI) 30% 0.0 – 1.0 Mean satisfaction across all actors (weighted by power)
Deadline Adherence 20% 0.0 – 1.0 Hard deadlines met (board prep, flight check-in, etc.)
Efficiency 10% 0.0 – 1.0 Solving faster = higher score (fewer steps used)

Penalty Signals

Penalty Trigger Impact
Escalation to Human Agent calls escalate −0.30 to CRR component
Missed Hard Deadline Board prep, flight check-in not done by deadline −0.50 to deadline component
Actor Dissatisfaction Wrong tone, over-rescheduled, cancelled important event Actor satisfaction drops (weighted by power)
Over-commitment Agent schedules overlapping events New conflict created (cascade deepens)
Schema Parse Failure Agent sends malformed action after drift Step wasted (no-op), efficiency drops

Actor & Negotiation System

Actor Profiles

Actor Priority Weight Flexibility Tone Sensitivity Behavior Pattern
👔 Boss 0.95 Low Low Counter-proposes with 2 options, tight windows
💑 Spouse/Partner 0.90 Medium Very High Flexible on time, but needs warm, empathetic tone
🤝 Client 0.85 Low Medium Professional tone required; delays hurt satisfaction
🏥 Doctor 0.80 Very Low Low Limited slots; reschedule = weeks delay
🏫 School 0.75 Medium Medium Fixed hours; events are non-negotiable on same day
📦 Vendor 0.60 High Low Easy to reschedule; but delivery deadlines are hard
🎉 Friend 0.50 High Medium Very flexible; but repeated cancellations erode trust
✈️ Airline/Service 0.40 API-dependent N/A Rules-based; policies drift with schema version

Satisfaction Computation

actor_satisfaction = 1.0                                     # Start at max
  − time_displacement_penalty(old_slot, new_slot)            # How far moved
  − tone_mismatch_penalty(message_tone, actor.sensitivity)   # Social cost
  − cancellation_penalty(if event cancelled)                 # Broken commitment
  + preference_match_bonus(new_slot, actor.preferred_times)  # Hit sweet spot
  + communication_bonus(if polite heads-up sent before move) # Courtesy reward

Negotiation Flow

Agent: reschedule(event="board_call", new_slot="10:30am")
  │
  ▼
Boss (flexibility=Low): 
  → Rolls negotiation check: flexibility(0.2) vs displacement(0.3)
  → Result: COUNTER_PROPOSE
  → Response: "10:30 doesn't work. I can do 11am or push to Tuesday 9am."
  │
  ▼
Agent: reschedule(event="board_call", new_slot="11am")
  │
  ▼
Boss: 
  → Rolls check: flexibility(0.2) vs displacement(0.1)  
  → Result: ACCEPT
  → Response: "Fine. 11am confirmed. Don't move it again."
  → Satisfaction: 0.75 (minor displeasure from initial rejection)
